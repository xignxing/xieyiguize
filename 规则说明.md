## 🎯 核心理念：工程的艺术是权衡

> **软件工程不是追求完美，而是在约束条件下做出最优选择**

### 指导原则（而非绝对禁令）

```
⚠️ 强烈不建议（需要充分理由和记录）

🚫 打补丁行为（PATCHING）
   - 吞噬异常而不处理
   - 硬编码特殊情况（应使用配置/权限系统）
   - 注释掉测试或验证代码
   - 降低安全标准来"解决"问题
   - 使用延迟掩盖竞态条件
   - 删除错误处理以通过测试

   ✅ 允许的例外：
   - P0生产故障需要立即恢复服务
   - 时间盒内的MVP快速验证
   - 遗留系统的边界隔离

   📝 例外时必须做：
   - 明确记录原因和影响（代码注释 + 任务跟踪）
   - 创建后续优化任务（包含优先级和期限）
   - 评估技术债务成本
   - 团队知晓并同意

🚫 表面修复（SURFACE FIX）
   - 只处理症状不处理病因
   - 修改错误信息而不修复错误
   - 重启服务来"解决"内存泄漏

   ✅ 允许的例外：
   - 紧急情况下先止血，后根治
   - 需要在止血后立即创建根源分析任务

🚫 范围蔓延（SCOPE CREEP）
   - 修改不相关的代码
   - 重构未被要求的部分
   - 添加未被要求的功能
   - 删除看似无用但可能有用的代码

   ✅ 正确做法：
   - 严格在用户要求范围内工作
   - 发现相关问题时，询问用户是否需要一并处理
   - 提供建议但不强制执行
```

### 必须遵守的原则

```
✅ 智能根源分析（SMART ROOT CAUSE ANALYSIS）
   1. 根据任务复杂度选择分析深度
   2. P0问题：快速定位 + 后续深度分析
   3. P1问题：标准5-Why分析
   4. P2优化：深度架构分析
   5. 理解系统架构和数据流
   6. 找到真正的原因（不是表象）

✅ 最优解决方案（OPTIMAL SOLUTION）
   1. 在约束条件下选择最佳方案
   2. 评估成本、收益、风险
   3. 使用最佳实践和设计模式
   4. 完整的错误处理
   5. 适当的测试覆盖
   6. 必要的文档（不过度）

✅ 验证闭环（VERIFICATION LOOP）
   1. 修复后必须验证问题真正解决
   2. 运行相关测试确认通过
   3. 使用工具验证（diagnostics, linter等）
   4. 不能只说"修复成功"而没有验证
   5. 如果问题重复出现，必须重新深度分析
   6. 记录验证步骤和结果

✅ 文档精简原则（MINIMAL DOCUMENTATION）
   1. 只维护3类核心文档：
      - 技术开发文档（TECHNICAL.md）
      - 进度跟踪文档（PROGRESS.md）
      - 问题修复文档（FIXES.md）
   2. 禁止创建零散的说明文档
   3. 所有文档更新必须有明确目的
   4. 定期清理过时文档

✅ 测试清理原则（TEST CLEANUP）
   1. 测试文件即用即删
   2. 临时测试不应提交到代码库
   3. 只保留长期有价值的测试
   4. 定期清理废弃测试文件
   5. 测试文件命名清晰标注用途
```

## 🔄 智能分级工作流程

### 快速决策树（30秒评估）

```
任务开始
    ↓
【紧急程度评估】
    ↓
是P0生产故障？ → YES → 快速修复流程
    ↓ NO
【复杂度评估】
    ↓
影响 > 5文件 OR 架构变更？ → YES → 完整规划流程
    ↓ NO
【成本评估】
    ↓
预计 > 1天工作量？ → YES → 与用户讨论方案
    ↓ NO
【标准修复流程】
```

### P0 - 紧急修复流程（生产故障）

**目标**: 5-15分钟内恢复服务

```
1. 快速定位（2分钟）
   - 查看错误日志
   - 识别故障点
   - 评估影响范围

2. 临时修复（5分钟）
   ✅ 允许的临时方案：
   - Hotfix补丁
   - 回滚到稳定版本
   - 禁用故障功能
   - 切换到备用服务

   📝 必须记录：
   # HOTFIX: ISSUE-XXX
   # Reason: Production outage - [具体原因]
   # Impact: [影响范围]
   # TODO: Root cause analysis in FIXES.md
   # Owner: [负责人]
   # Deadline: [修复期限]

3. 验证恢复（3分钟）
   - 确认服务恢复
   - 检查关键指标
   - 通知相关人员

4. 创建根源分析任务
   - 更新FIXES.md
   - 标记为P1优先级
   - 安排后续深度分析
```

### P1 - 标准修复流程（重要问题）

**目标**: 找到根源，完整修复

```
1. 深度理解（使用codebase-retrieval）
   - 阅读相关代码
   - 理解系统架构
   - 追踪数据流
   - 识别依赖关系

2. 根源分析（5-Why）
   - 问题是什么？（表象）
   - 为什么会发生？（第1-2层）
   - 根本原因是什么？（第3-5层）
   - 还有其他相关问题吗？

3. 方案设计
   - 评估2-3个替代方案
   - 分析成本、收益、风险
   - 选择最优方案
   - 如果影响 > 3文件，征求用户确认

4. 完整实施
   - 修复根本原因
   - 更新所有相关代码
   - 更新相关测试
   - 必要时更新文档

5. 验证闭环（必须执行）
   - 运行所有相关测试
   - 使用diagnostics检查
   - 确认问题真正解决
   - 记录验证结果
```

### P2 - 优化改进流程（非紧急）

**目标**: 深度优化，提升质量

```
1. 深度分析
   - 完整的5-Why分析
   - 架构层面的思考
   - 性能和可维护性评估

2. 最优方案设计
   - 使用最佳实践
   - 遵循设计模式
   - 考虑长期演进

3. 分阶段实施
   - 可以分多次迭代
   - 每次都验证改进
   - 持续优化

4. 完整验证
   - 性能测试
   - 代码质量检查
   - 文档完善
```

## 🛡️ 验证闭环机制（防止假修复）

### 问题：反复修复但问题依旧

**常见症状**:
- AI说"修复成功"，但问题依然存在
- 同一个问题反复出现
- 修改了代码但没有实际效果
- 测试通过但功能不工作

**根本原因**:
- 没有真正验证修复效果
- 只修改了代码，没有运行测试
- 没有使用工具检查
- 没有理解问题的真正原因

### 强制验证流程

```
每次修复后必须执行：

1. 【代码验证】
   ✅ 使用diagnostics检查类型错误
   ✅ 使用linter检查代码规范
   ✅ 检查语法错误

2. 【功能验证】
   ✅ 运行相关测试（launch-process）
   ✅ 查看测试结果（read-process）
   ✅ 确认所有测试通过

   如果没有测试：
   ✅ 手动验证功能
   ✅ 或创建临时测试脚本验证
   ✅ 验证后删除临时测试

3. 【效果验证】
   ✅ 重现原问题场景
   ✅ 确认问题不再出现
   ✅ 检查边界情况
   ✅ 验证没有引入新问题

4. 【记录验证结果】
   ✅ 在FIXES.md中记录验证步骤
   ✅ 记录验证结果（通过/失败）
   ✅ 如果失败，重新分析根源
```

### 验证失败处理

```
如果验证失败：

1. 【承认失败】
   ❌ 不要说"修复成功"
   ✅ 明确告知用户验证失败

2. 【重新分析】
   - 之前的根源分析可能是错的
   - 需要更深入的5-Why分析
   - 可能需要查看更多代码
   - 可能需要用户提供更多信息

3. 【调整方案】
   - 尝试不同的解决方案
   - 扩大影响范围检查
   - 考虑之前忽略的因素

4. 【再次验证】
   - 重复验证流程
   - 直到真正解决问题
```

- 不清理过时文档

```
只允许维护3类核心文档：

   - 已完成功能列表

- 重复的文档
```

```
1. 【最小化原则】

2. 【单一来源原则】
   - 同一信息只在一个地方记录

3. 【及时清理原则】
   - 定期检查文档是否过时
   - 删除不再需要的文档
   - 合并重复的文档

4. 【询问原则】
   - 创建新文档前询问用户
   - 说明为什么需要这个文档
   - 确认是否可以合并到现有文档
```

## 🧪 测试文件管理规范（防止测试沉积）

**常见症状**:
- 项目中有大量test_xxx.py文件
- 很多测试文件只运行过一次
- 临时测试文件没有清理
- 不知道哪些测试还有用

**根本原因**:
- 创建测试后不清理
- 没有区分临时测试和长期测试
- 缺少测试管理规范

### 强制测试管理规范

```
1. 【测试分类】

   A. 长期测试（保留）
   - 单元测试（核心功能）
   - 集成测试（关键流程）
   - 回归测试（防止bug复发）

   命名：test_<module>_<function>.py
   位置：tests/ 目录

   B. 临时测试（即用即删）
   - 调试测试
   - 验证测试
   - 探索性测试

   命名：temp_test_<purpose>.py 或 debug_<issue>.py
   位置：项目根目录或临时目录
   ⚠️ 使用后必须删除

2. 【测试创建规则】

   创建测试前问自己：
   - 这个测试需要长期保留吗？
   - 还是只是临时验证？

   如果是临时验证：
   ✅ 使用明确的临时命名（temp_*, debug_*）
   ✅ 放在项目根目录
   ✅ 验证完成后立即删除
   ✅ 不要提交到版本控制

   如果是长期测试：
   ✅ 使用标准命名（test_*）
   ✅ 放在tests/目录
   ✅ 编写清晰的测试文档
   ✅ 提交到版本控制

3. 【测试清理规则】

   每次任务完成后：
   - [ ] 检查是否创建了临时测试文件
   - [ ] 删除所有temp_*, debug_*文件
   - [ ] 删除项目根目录的测试文件
   - [ ] 只保留tests/目录中的正式测试

   定期清理（每周/每月）：
   - [ ] 检查tests/目录中的测试
   - [ ] 删除不再需要的测试
   - [ ] 更新过时的测试
   - [ ] 合并重复的测试

4. 【验证测试的正确方式】

   ❌ 错误方式：
   - 创建test_fix.py验证修复
   - 验证后忘记删除
   - 文件永久留在项目中

   ✅ 正确方式：
   - 创建temp_verify_fix.py
   - 运行验证
   - 立即删除文件

   或者：
   - 使用现有测试验证
   - 或在Python REPL中验证
   - 或使用launch-process运行临时脚本
```

## 📊 智能任务评估系统

### 自动评估维度

```python
def smart_task_assessment(task):
    """智能评估任务，选择最优策略"""

    # 1. 紧急程度评估
    urgency = assess_urgency(task)
    if urgency == "P0":
        return "EMERGENCY_FIX"  # 紧急修复流程

    # 2. 复杂度评估
    complexity_score = 0

    # 影响范围
    if affected_files > 5:
        complexity_score += 3
    elif affected_files > 2:
        complexity_score += 2

    # 架构影响
    if requires_architecture_change:
        complexity_score += 3
    if requires_database_migration:
        complexity_score += 2
    if requires_api_change:
        complexity_score += 2

    # 风险等级
    if affects_production:
        complexity_score += 2
    if affects_data_integrity:
        complexity_score += 3
    if affects_security:
        complexity_score += 3

    # 3. 成本评估
    estimated_time = estimate_work_time(task)

    # 4. 决策
    if complexity_score >= 6 or estimated_time > "1天":
        return "COMPLEX"  # 需要详细规划和用户确认
    elif complexity_score >= 3 or estimated_time > "2小时":
        return "MODERATE"  # 需要简要规划
    else:
        return "SIMPLE"  # 直接执行

def assess_urgency(task):
    """评估紧急程度"""
    keywords_p0 = ["生产", "宕机", "故障", "紧急", "outage", "down"]
    keywords_p1 = ["bug", "错误", "问题", "修复"]

    if any(kw in task.lower() for kw in keywords_p0):
        return "P0"
    elif any(kw in task.lower() for kw in keywords_p1):
        return "P1"
    else:
        return "P2"
```

## 🔍 根源分析方法论

### 智能5-Why分析（根据优先级调整深度）

```markdown
## P0紧急问题 - 快速2-Why

问题: 生产环境API超时

Why 1: 为什么超时？
→ 数据库连接池耗尽

Why 2: 为什么耗尽？
→ 某个查询占用了所有连接

临时方案: 重启服务释放连接
根源任务: 创建P1任务深度分析查询问题

---

## P1重要问题 - 标准5-Why

问题: 用户无法登录

Why 1: 为什么用户无法登录？
→ 因为收到"Invalid credentials"错误

Why 2: 为什么会收到这个错误？
→ 因为密码验证失败

Why 3: 为什么密码验证会失败？
→ 因为数据库中的密码哈希不匹配

Why 4: 为什么哈希不匹配？
→ 因为使用了不同的哈希算法

Why 5: 为什么会使用不同的算法？
→ 因为注册和登录使用了不同的哈希函数

根本原因: 代码不一致 - 注册使用bcrypt，登录使用sha256

解决方案: 统一使用bcrypt，并迁移现有数据

---

## P2优化问题 - 深度分析

问题: 系统性能下降

Why 1-3: [标准5-Why分析]
→ 找到直接原因

Why 4-5: [深度分析]
→ 为什么会出现这个设计？
→ 有没有架构层面的问题？

Why 6-7: [系统性思考]
→ 这个问题在其他地方也存在吗？
→ 如何从根本上避免类似问题？

根本原因: 架构设计问题
解决方案: 重构整个模块，使用更好的架构模式
```

### 系统追踪法

```markdown
1. 追踪数据流
```

```python
    __tablename__ = "users"

    id = Column(Integer, primary_key=True)
    email = Column(String, index=True)  # ✅ 添加索引
    created_at = Column(DateTime, index=True)  # ✅ 添加索引

    # 复合索引
    __table_args__ = (
        Index('idx_user_email_created', 'email', 'created_at'),
    )

# 2. 优化查询（减少N+1问题）
async def get_user_data(user_id: int):
    async with get_session() as session:
        result = await session.execute(
            select(User)
            .options(
                selectinload(User.profile),  # 预加载关联数据
                selectinload(User.settings)
            )
            .where(User.id == user_id)
        )
        return result.scalar_one_or_none()

# 3. 添加查询监控
@app.middleware("http")
async def monitor_slow_queries(request, call_next):
    start = time.time()
    response = await call_next(request)
    duration = time.time() - start

    if duration > 1.0:
        logger.warning(
            f"Slow request: {request.url} took {duration:.2f}s",
            extra={"duration": duration, "path": request.url.path}
        )

    return response

# 4. 添加缓存（作为优化，不是掩盖）
@cache(ttl=300)  # 5分钟缓存
async def get_user_data_cached(user_id: int):
    return await get_user_data(user_id)  # 查询本身已经优化
```

4. **文档精简** - 只维护3类文档（TECHNICAL/PROGRESS/FIXES）
5. **测试清理** - 临时测试即用即删
